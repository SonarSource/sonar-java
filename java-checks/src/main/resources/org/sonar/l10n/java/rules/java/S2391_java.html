If the suite() method in a JUnit 3 TestCase is not declared correctly, it will not be used. Such a method must be named "suite", have no arguments, be public static, and must return either a junit.framework.Test or a junit.framework.TestSuite.

Similarly, setUp() and tearDown() methods that aren't properly capitalized will also be ignored.

Using JUnit4 and JUnit5, however, the suite() methods become obsolete, has the test suite mechanism has been changed to rely on class annotations. setUp() and tearDown() methods, on their side, needs to be correctly annotated by the equivalent annotation in order to preserve the same behavior.

Noncompliant Code Example

  • JUnit3: 
Test suite() { ... }  // Noncompliant; must be public static
public static boolean suite() { ... }  // Noncompliant; wrong return type
public static Test suit() { ... }  // Noncompliant; typo in method name
public static Test suite(int count) { ... } // Noncompliant; must be no-arg

public void setup() { ... } // Noncompliant; should be setUp
public void tearDwon() { ... }  // Noncompliant; should be tearDown
  • JUnit4: 
public static Test suite() { ... }  // Noncompliant; does not make sense anymore

public void setUp() { ... } // Noncompliant; should be annotated with @Before
public void tearDown() { ... }  // Noncompliant; should be annotated with @After
  • JUnit5: 
public static Test suite() { ... }  // Noncompliant; does not make sense anymore

public void setUp() { ... } // Noncompliant; should be annotated with @BeforeEach
public void tearDown() { ... }  // Noncompliant; should be annotated with @AfterEach

Compliant Solution

  • JUnit3: 
public static Test suite() { ... }
public void setUp() { ... }
public void tearDown() { ... }
  • JUnit4: 
@Before
public void setUp() { ... }

@After
public void tearDown() { ... }
  • JUnit5: 
@BeforeEach
void setUp() { ... }

@AfterEach
void tearDown() { ... }
