This rule raises an issue when a string literal that is not a valid time zone identifier is passed to methods that accept time zone identifier arguments.

In Java, this specifically applies to the ZoneId.of() method.

Why is this an issue?

Standard time zone libraries provide methods that create time zone objects from string identifiers. When using these methods, the provided string must be a valid time zone identifier recognized by the library.

Valid time zone identifiers typically include:

  • IANA Time Zone Database names (e.g., "America/New_York", "Europe/Paris")
  • Fixed offsets in the format "+HH:MM" or "-HH:MM" (e.g., "+05:30", "-08:00")
  • Special identifiers like "UTC", "GMT", or "Z" (which is short for UTC)
  • Zone offset IDs like "UTC+1" or "GMT-5"

If you pass an invalid string to a time zone creation method, the library throws a runtime exception. This can happen due to:

  • Typos in zone identifier strings
  • Using three-letter abbreviations that are not valid zone IDs (like "PST", "EST")
  • Using deprecated or obsolete zone identifiers (like "US/Pacific")
  • Completely invalid strings

Three-letter time zone abbreviations are particularly problematic because they are ambiguous. For example, "CST" could mean Central Standard Time (North America), China Standard Time, or Cuba Standard Time. Modern time zone APIs require full IANA zone names to avoid this ambiguity.

Deprecated zone IDs like "US/Pacific" or "US/Eastern" should also be avoided. While they may still work in some library versions, they are not part of the standard IANA Time Zone Database and may be removed in future versions.

In Java, this refers to the java.time.ZoneId class and its static method ZoneId.of(String zoneId), which throws a ZoneRulesException when given an invalid zone identifier.

What is the potential impact?

When an invalid string identifier is used to look up a system resource or configuration value, the application will throw a runtime exception indicating the resource cannot be found. This exception occurs when the code path is executed, which could be:

  • During application startup, causing the application to fail to launch
  • During request handling, resulting in failed user requests and poor user experience
  • In scheduled tasks, causing background jobs to fail silently

Because this error occurs at runtime rather than compile time, it may not be discovered during development and could surface in production environments, leading to application downtime or data processing failures.

How to fix it

Replace invalid zone ID strings with valid IANA Time Zone Database identifiers. If you were using a three-letter abbreviation, replace it with the full zone name from the IANA database. If you were using a deprecated zone ID, update it to the current standard identifier.

Code examples

Noncompliant code example

ZoneId zone1 = ZoneId.of("InvalidZone");  // Noncompliant
ZoneId zone2 = ZoneId.of("PST");          // Noncompliant
ZoneId zone3 = ZoneId.of("US/Pacific");   // Noncompliant

Compliant solution

ZoneId zone1 = ZoneId.of("Europe/Paris");         // Compliant: valid IANA zone ID
ZoneId zone2 = ZoneId.of("America/Los_Angeles"); // Compliant: full name instead of PST
ZoneId zone3 = ZoneId.of("America/Los_Angeles"); // Compliant: current standard ID
