HTTP request paths used in security checks without normalization can allow attackers to bypass authentication and authorization by inserting extra slashes in URLs.

Why is this an issue?

Web applications often implement security policies based on URL paths. A critical vulnerability exists when these security checks compare request paths directly without normalization. URL paths can contain multiple consecutive forward slashes (e.g., //admin, /admin//), which many frameworks preserve in the raw request path.

When security logic performs string comparisons on non-normalized paths, it may fail to match intended patterns, allowing attackers to bypass security controls through URL manipulation.

How to fix it

Normalize the request path by collapsing consecutive slashes before performing security checks. Use a regular expression to replace multiple slashes with a single slash.

Code examples

Noncompliant code example

String path = requestContext.getUriInfo().getPath();
if (path.startsWith("/admin")) { // Noncompliant
    checkAdminPermission();
}

Compliant solution

String path = requestContext.getUriInfo().getPath();
String normalizedPath = path.replaceAll("/+", "/");
if (normalizedPath.startsWith("/admin")) {
    checkAdminPermission();
}
