Using .from() to instantiate LocalDate, LocalTime, YearMonth, or Year from a current time expression is unnecessarily verbose when a direct .now() or .now(ZoneId) factory method is available.

Why is this an issue?

The following patterns create intermediate objects that are immediately discarded:

  • Type.from(Instant.now()) → use Type.now()
  • Type.from(Instant.now().atZone(zone)) → use Type.now(zone)
  • Type.from(ZonedDateTime.now(zone)) → use Type.now(zone)
  • Type.from(OffsetDateTime.now(zone)) → use Type.now(zone)
  • YearMonth.from(LocalDate.now(zone)) → use YearMonth.now(zone)
  • Year.from(LocalDate.now(zone)) or Year.from(YearMonth.now(zone)) → use Year.now(zone)

Calling .now() directly on the target type is more concise, avoids allocating objects whose sole purpose is immediate conversion, and makes the intent clearer.

Note that .from() is still appropriate when the argument is an existing temporal variable (not a fresh now() call), for example LocalDate.from(existingInstant.atZone(zone)).

How to fix it

Replace Type.from(nowExpression) with Type.now() or Type.now(zone) directly on the target type.

Code examples

Noncompliant code example

import java.time.*;

LocalDate date1 = LocalDate.from(Instant.now());                          // Noncompliant
LocalDate date2 = LocalDate.from(Instant.now().atZone(ZoneId.of("UTC"))); // Noncompliant
LocalDate date3 = LocalDate.from(ZonedDateTime.now(ZoneId.of("UTC")));    // Noncompliant
LocalDate date4 = LocalDate.from(OffsetDateTime.now(ZoneId.of("UTC")));   // Noncompliant
Year      year  = Year.from(YearMonth.now(ZoneId.of("UTC")));             // Noncompliant

Compliant solution

import java.time.*;

LocalDate date1 = LocalDate.now();
LocalDate date2 = LocalDate.now(ZoneId.of("UTC"));
LocalDate date3 = LocalDate.now(ZoneId.of("UTC"));
LocalDate date4 = LocalDate.now(ZoneId.of("UTC"));
Year      year  = Year.now(ZoneId.of("UTC"));
